# Exercise 3 — Production Jobs, Slim CI & Exposures

**Time: ~90 minutes**

## Goal
Configure dbt Cloud for production: a scheduled daily job, a Slim CI job that runs only modified models on every pull request, and Exposures to document downstream consumers.

---

## The Three Job Types

dbt Cloud has three job types, each with a different purpose:

**Deploy Job** — runs on a schedule or manually. Used for the daily production build that rebuilds the full project and generates artifacts. This is the baseline everything else depends on.

**CI Job (pre-merge)** — triggers automatically when a Pull Request is opened. Runs only modified models (`state:modified+`) and writes to a temporary schema (`dbt_cloud_pr_<job_id>_<pr_id>_<schema>`) that dbt Cloud creates and deletes automatically. Purpose: validate that the change doesn't break anything before it reaches `main`.

**Deploy Job with merge trigger (post-merge)** — triggers when a PR is merged to `main`. Also runs only modified models but writes to the real production schemas. Purpose: deploy changes immediately without waiting for the next scheduled run.

Together they give you: **validate on PR → deploy on merge → rebuild everything nightly**.

## Part A — Fix generate_schema_name

The default `generate_schema_name` macro in this project does not include the `dbt_cloud_pr_` prefix, which means CI job runs would write to your real production schemas instead of isolated temporary ones. Fix it before configuring any jobs.

This macro already exists in your project from Day 3. Open `macros/generate_schema_name.sql` and replace its contents with:

```jinja-sql
{% macro generate_schema_name(custom_schema_name, node) -%}
    {%- set default_schema = target.schema -%}
    {%- if custom_schema_name is none -%}
        {{ default_schema }}
    {%- elif env_var('DBT_CLOUD_PR_ID', '') != '' -%}
        dbt_cloud_pr_{{ env_var('DBT_CLOUD_JOB_ID') }}_{{ env_var('DBT_CLOUD_PR_ID') }}_{{ custom_schema_name | trim }}
    {%- else -%}
        {{ custom_schema_name | trim }}
    {%- endif -%}
{%- endmacro %}
```

What this does: dbt Cloud automatically sets `DBT_CLOUD_PR_ID` and `DBT_CLOUD_JOB_ID` environment variables on every CI run. The macro detects them and prefixes the schema accordingly. In development and production runs those variables don't exist, so the macro behaves exactly as before.

Commit and push this change before configuring any jobs.

## Part B — Environments vs Jobs

Before configuring anything, understand the relationship between the two core concepts:

**Environments** define *where* dbt runs — which database, schema, warehouse, credentials, and dbt version. You configure them once and jobs reference them. You need at least two: Development (already set up on Day 1) and Production.

**Jobs** define *what* runs and *when* — the commands, schedule or trigger, and which environment to use. Multiple jobs can share the same environment.

The artifact flow between jobs is what makes deferral work:

```
Production — Daily Build
  └── generates manifest.json + run_results.json
        └── stored as artifacts in dbt Cloud
              └── CI Job reads manifest.json to diff what changed
              └── CD Job reads manifest.json to diff what changed
```

When you enable "Defer to a previous run state" in a CI or CD job, dbt Cloud downloads the manifest from the deferred job and uses it to compute `state:modified+`. Without a completed production run there is no manifest to diff against, so CI/CD cannot work.

In dbt Cloud you need two environments — development (already set up on Day 1) and production.

Go to **Deploy → Environments → New Environment** and fill in:

| Field | Value |
|-------|-------|
| Name | `Production` |
| Environment type | `Deployment` |
| dbt version | same as Development |
| Default schema | leave blank (comes from `dbt_project.yml`) |
| Credentials | `TRANSFORMER` role, `DBT_WH` |

## Part D — Daily Production Job

Go to **Deploy → Jobs → Create Job → Deploy Job**:

**Job name:** `Production — Daily Build`  
**Environment:** Production  
**Commands:**
```
dbt source freshness
dbt build
```

**Schedule:** enable scheduling → Cron → `0 6 * * *` (every day at 06:00 UTC)  
**Advanced settings:** tick *Generate docs on run*

Save, then click **Run Now** to test it manually.

After it finishes:
- Click the run to inspect the run log — which models ran, in which order, how long each took
- Open the **Artifacts** tab and find the slowest model in `run_results.json`
- Open the automatically generated docs site from the **Artifacts** tab

**Important:** The production job must complete at least once before Slim CI (Part E) can work — Slim CI diffs against the production manifest generated by this job.

## Part E — Slim CI Job (Pre-merge)

Slim CI runs **only the models that changed** in a pull request instead of rebuilding the whole project. This makes CI fast enough to run on every PR.

Go to **Deploy → Jobs → Create Job → CI Job**:

**Job name:** `CI — Pull Request Check`  
**Environment:** Production

Under **Execution settings**, enable **Defer to a previous run state** and select the `Production — Daily Build` job. dbt Cloud will automatically download that job's manifest and use it to detect what changed — you do not write `--state` flags manually.

**Commands:**
```
dbt build --select state:modified+
```

**Triggers:** *Run on pull request* is enabled by default for CI jobs.

What this does:
- `state:modified+` — builds models that changed in this branch + all their downstream dependents
- **Defer** — for models *not* in scope, dbt Cloud borrows the compiled results from the last successful production run so their downstream tests still work

**Test it:**

In the dbt Cloud IDE, create a new branch:

1. Click the branch name in the top left → **Create branch** → name it `feat/add-order-year`
2. Open `models/gold/gold_orders.sql` and add `YEAR(order_date) AS order_year` to the SELECT
3. Commit the change
4. Open a Pull Request on GitHub

Watch the CI job trigger automatically in **Deploy → Jobs → CI — Pull Request Check**.

**Questions:**
- Which models ran? Which were deferred?
- How long did the CI run take compared to a full `dbt build`?
- What would happen if you deleted the `--select state:modified+` flag?

## Part F — CD Job (Post-merge)

Go to **Deploy → Jobs → Create Job → Deploy Job**:

**Job name:** `CD — Deploy on Merge`  
**Environment:** Production  
**Commands:**
```
dbt build --select state:modified+
```

Under **Execution settings**, enable **Defer to a previous run state** and select `Production — Daily Build`.

Under **Triggers**, disable the schedule and enable **Continuous deployment** → *Run when changes are merged into main*.

This job deploys modified models to production the moment a PR merges — no waiting for the 6am cron. The next daily build will include those models as part of the full rebuild.

## Part G — Monitoring Runs

After the daily job runs, go to **Deploy → Run History**. Every run shows:

- Status (success, error, cancelled)
- Duration and which step failed
- The full command log per model — click any model to see the SQL that executed and the error if it failed

In the **Artifacts** tab of a completed run you have access to:

| File | What it contains |
|------|-----------------|
| `manifest.json` | Full DAG — every node, its dependencies, config, and compiled SQL |
| `run_results.json` | Execution status, timing, and row count per model |
| `sources.json` | Source freshness results |
| `catalog.json` | Column metadata from the warehouse (generated by `dbt docs generate`) |

`run_results.json` is useful for finding slow models — each node has an `execution_time` in seconds. Use it to identify candidates for incremental materialization.

**Job retries:** in job settings you can configure automatic retries on failure. Useful for transient Snowflake errors (warehouse suspend, timeout). Set to 1 retry maximum — if a job fails twice it's a real error, not a transient one.

## Part H — Environment Variables

Jobs can have custom environment variables under **Advanced settings → Environment variables**. These are available inside your dbt project as `env_var('MY_VAR')`.

Common uses:
- Pass a `DBT_TARGET` flag to control behavior per environment
- Store API keys for custom packages that call external services
- Override a variable without changing `dbt_project.yml`

Variables set at the job level override variables set at the environment level, which override account-level variables. dbt Cloud never exposes secret variables in logs.

## Part I — Exposures

Exposures document what consumes your dbt models — dashboards, reports, ML models. They close the lineage graph from raw source all the way to end consumers.

Create `models/exposures.yml`:

```yaml
version: 2

exposures:

  - name: orders_dashboard
    label: "Orders & Revenue Dashboard"
    type: dashboard
    maturity: high
    url: https://your-bi-tool.com/dashboards/orders
    description: >
      Main revenue operations dashboard. Shows daily order volume,
      revenue by channel, and fulfilment status. Used by Operations
      and Finance every morning.
    depends_on:
      - ref('gold_orders')
      - ref('gold_customers')
    owner:
      name: "Data Team"
      email: "data@example.com"

  - name: customer_360_report
    label: "Customer 360 Report"
    type: analysis
    maturity: medium
    description: >
      Customer lifetime value and segmentation. Feeds the CRM system
      for marketing automation.
    depends_on:
      - ref('gold_customers')
    owner:
      name: "Marketing Analytics"
      email: "marketing-analytics@example.com"
```

Commit and run in the IDE:

```bash
dbt docs generate
```

In the dbt Cloud docs site, verify:
- Exposures appear as terminal nodes in the DAG
- Clicking an exposure shows owner, description, and linked models
- `dbt run --select +exposure:orders_dashboard` runs only the models that feed it

**Note:** You will add a third exposure (`procurement_dashboard`) in the Capstone exercise once those models exist.
