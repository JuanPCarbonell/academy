# Exercise 3 — Production Jobs, Slim CI & Exposures

**Time: ~90 minutes**

## Goal
Configure dbt Cloud for production: a scheduled daily job, a Slim CI job that runs only modified models on every pull request, and Exposures to document downstream consumers.

---

## Part A — Configure the Production Environment

In dbt Cloud you need two environments — development (already set up on Day 1) and production.

Go to **Deploy → Environments → New Environment** and fill in:

| Field | Value |
|-------|-------|
| Name | `Production` |
| Environment type | `Deployment` |
| dbt version | same as Development |
| Default schema | leave blank (comes from `dbt_project.yml`) |
| Credentials | `TRANSFORMER` role, `DBT_WH` |

---

## Part B — Daily Production Job

Go to **Deploy → Jobs → Create Job → Deploy Job**:

**Job name:** `Production — Daily Build`  
**Environment:** Production  
**Commands:**
```
dbt source freshness
dbt build
```

**Schedule:** enable scheduling → Cron → `0 6 * * *` (every day at 06:00 UTC)  
**Advanced settings:** tick *Generate docs on run*

Save, then click **Run Now** to test it manually.

After it finishes:
- Click the run to inspect the run log — which models ran, in which order, how long each took
- Open the **Artifacts** tab and find the slowest model in `run_results.json`
- Open the automatically generated docs site from the **Artifacts** tab

**Important:** The production job must complete at least once before Slim CI (Part C) can work — Slim CI diffs against the production manifest generated by this job.

---

## Part C — Slim CI Job

Slim CI runs **only the models that changed** in a pull request instead of rebuilding the whole project. This makes CI fast enough to run on every PR.

Go to **Deploy → Jobs → Create Job → CI Job**:

**Job name:** `CI — Pull Request Check`  
**Environment:** Production

Under **Execution settings**, enable **Defer to a previous run state** and select the `Production — Daily Build` job. dbt Cloud will automatically download that job's manifest and use it to detect what changed — you do not write `--state` flags manually.

**Commands:**
```
dbt build --select state:modified+
```

**Triggers:** *Run on pull request* is enabled by default for CI jobs.

What this does:
- `state:modified+` — builds models that changed in this branch + all their downstream dependents
- **Defer** — for models *not* in scope, dbt Cloud borrows the compiled results from the last successful production run so their downstream tests still work

**Test it:**

In the dbt Cloud IDE, create a new branch:

1. Click the branch name in the top left → **Create branch** → name it `feat/add-order-year`
2. Open `models/gold/gold_orders.sql` and add `YEAR(order_date) AS order_year` to the SELECT
3. Commit the change
4. Open a Pull Request on GitHub

Watch the CI job trigger automatically in **Deploy → Jobs → CI — Pull Request Check**.

**Questions:**
- Which models ran? Which were deferred?
- How long did the CI run take compared to a full `dbt build`?
- What would happen if you deleted the `--select state:modified+` flag?

## Part D — Exposures

Exposures document what consumes your dbt models — dashboards, reports, ML models. They close the lineage graph from raw source all the way to end consumers.

Create `models/exposures.yml`:

```yaml
version: 2

exposures:

  - name: orders_dashboard
    label: "Orders & Revenue Dashboard"
    type: dashboard
    maturity: high
    url: https://your-bi-tool.com/dashboards/orders
    description: >
      Main revenue operations dashboard. Shows daily order volume,
      revenue by channel, and fulfilment status. Used by Operations
      and Finance every morning.
    depends_on:
      - ref('gold_orders')
      - ref('gold_customers')
    owner:
      name: "Data Team"
      email: "data@example.com"

  - name: customer_360_report
    label: "Customer 360 Report"
    type: analysis
    maturity: medium
    description: >
      Customer lifetime value and segmentation. Feeds the CRM system
      for marketing automation.
    depends_on:
      - ref('gold_customers')
    owner:
      name: "Marketing Analytics"
      email: "marketing-analytics@example.com"
```

Commit and run in the IDE:

```bash
dbt docs generate
```

In the dbt Cloud docs site, verify:
- Exposures appear as terminal nodes in the DAG
- Clicking an exposure shows owner, description, and linked models
- `dbt run --select +exposure:orders_dashboard` runs only the models that feed it

**Note:** You will add a third exposure (`procurement_dashboard`) in the Capstone exercise once those models exist.

## Part E — Notifications

In **Account Settings → Notifications**, configure email notifications for job failures. Set to notify on failure only — success notifications create noise and get ignored.

## ✅ Success Criteria

- [ ] Production environment configured in dbt Cloud
- [ ] Daily production job runs successfully (green) and generates artifacts
- [ ] CI job configured with deferral to the production job — no `--state` path written manually
- [ ] CI job triggers automatically on a pull request
- [ ] Run log shows only modified models + downstream ran; others were deferred
- [ ] `exposures.yml` with 2 exposures — they appear as terminal nodes in the DAG
- [ ] Email notification configured for failures
