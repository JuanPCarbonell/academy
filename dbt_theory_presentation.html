Context

How We Got Here

From spreadsheets to the modern data stack

1980s
Spreadsheets & CSV Excel, Lotus 1-2-3 — analysts own the data, no shared infrastructure
1990s
Relational Databases & On-prem DWH Oracle, SQL Server, Teradata — SQL emerges, but ETL pipelines are expensive and slow to change
2000s
Big Data Era Hadoop, MapReduce — scale at last, but requires specialized engineering and Java. Analysts locked out.
2010–15
Cloud Data Warehouses Redshift (2012), BigQuery (2010), Snowflake (2012) — columnar, elastic, SQL-first. Storage and compute separated.
2016
dbt is born Fishtown Analytics releases dbt — "the T in ELT". Load raw data first, transform inside the warehouse with SQL + software engineering practices.
2020s
Modern Data Stack Fivetran → Snowflake → dbt → Looker/Tableau. Each layer does one thing well. 100k+ companies run dbt in production.
💡 The shift to cloud DWH made it cheap and fast to load raw data first. That made ELT — and dbt — possible.

Version Control

The Problem Before Git

You've already lived this. Probably with spreadsheets.

The "final" file problem
analysis_v1.sql
analysis_v2_FINAL.sql
analysis_v2_FINAL_revised.sql
analysis_v2_FINAL_revised_USE_THIS.sql

Sound familiar? No one knows which is current. No one knows what changed between versions. If something breaks, there's no way to go back safely.
The collaboration problem
Two people edit the same SQL file. One saves over the other's work. There's no merge — just lost changes and broken pipelines. Email attachments and shared drives are not a solution.

Version control solves all of this by answering four questions about every file, at every moment:

What changed?
Every line added, removed, or modified is recorded with a diff.
Who changed it?
Every change is signed by the author. Accountability is built in.
When?
A full timestamp history. You can see exactly what the code looked like at any point in time.
Why?
Every change has a commit message explaining the intent. Future-you will be grateful.
Version Control

Git — How It Works

Built in 2005 by Linus Torvalds in 10 days to manage the Linux kernel

Before Git, the Linux kernel used BitKeeper — a proprietary tool. When the license was revoked in 2005, Torvalds built Git from scratch in two weeks. His goals: fast, distributed, and impossible to corrupt.

Repository (repo) — a folder tracked by Git. Contains all your files plus the full history of every change ever made.
Commit — a snapshot of your files at a point in time. Like a save point in a game. Each commit has an ID, author, timestamp, and message.
Branch — a parallel version of the repo. Work on a feature in isolation without touching the main codebase. Cheap to create, safe to experiment in.
Merge — combines two branches. Git is smart about this — it applies only the differences, line by line.
💡 Distributed: every developer has a full copy of the entire repo history on their machine. No central server = no single point of failure.

The three stages of a file in Git:

1
Working directory
You edited the file. Git sees the change but isn't tracking it yet.
2
Staging area git add
You chose what to include in the next commit. Like putting items in a box before sealing it.
3
Repository git commit
The snapshot is sealed and added to the permanent history. This cannot be undone (only appended to).
🌿 Today there are over 100 million repositories on GitHub. Git is used by virtually every software team on the planet — and increasingly by data teams.

Core Concept

What is dbt?

dbt is an open-source transformation framework that lets
data analysts and engineers write modular SQL SELECT statements
that dbt compiles and runs against your data warehouse.

🔧
Just SQL
Write SELECT statements. dbt handles the CREATE OR REPLACE boilerplate.
🌐
Version Controlled
All transformations live in Git. Code review, history, rollbacks — standard dev workflow.
🧪
Tested & Documented
Built-in testing and auto-generated documentation as first-class features.
🎯 Key insight: dbt does the T in ELT. It only transforms data that is already in your warehouse — it doesn't move or load it.
Architecture

ETL vs ELT

Old way — ETL
① Extract
↓ Transform happens OUTSIDE the warehouse
(Python, Spark, custom ETL tools)
② Transform
↓ Only then load clean data
③ Load
⚠️Expensive to re-transform, slow iteration, requires separate compute.
Modern — ELT
① Extract
↓ Load raw data FIRST (Fivetran, Airbyte, etc.)
② Load (raw)
↓ Transform INSIDE the warehouse using SQL
③ Transform ← dbt lives here
✅Use warehouse compute power. Raw data is always preserved. Fast iteration.

Core Concept

Models — The Heart of dbt

A model is just a .sql file containing a single SELECT statement

-- models/bronze/bronze_tpch_orders.sql
-- That's it. Just a SELECT.

SELECT
    o_orderkey              AS order_id,
    o_custkey               AS customer_id,
    o_orderstatus           AS order_status,
    o_totalprice            AS total_price,
    o_orderdate             AS order_date
FROM {{ source('tpch', 'orders') }}
⚡ dbt compiles this into CREATE OR REPLACE TABLE / VIEW — you never write DDL.
Naming convention
bronze_ → bronze layer (clean, standardize raw data)
silver_ → silver layer (business logic, joins)
gold_ → gold layer (analytics-ready, aggregations)
Run commands
dbt run — run all models
dbt run --select bronze_tpch_orders — single model
dbt run --select bronze.* — by folder
dbt run --select +gold_orders — with deps
Sources

Sources — Raw Data Contracts

Declare where your raw data lives, then reference it by name

Declare in models/bronze/sources.yml

version: 2

sources:
  - name: tpch
    database: SNOWFLAKE_SAMPLE_DATA
    schema: TPCH_SF1
    tables:
      - name: orders
        description: "1.5M orders"

      - name: lineitem
        description: "6M line items"
        freshness: null   # read-only source

      - name: customer
        description: "150K customers"

Reference in models

-- ✅ Use source() — tracked by dbt
SELECT * FROM {{ source('tpch', 'orders') }}

-- ❌ Don't hardcode table names
SELECT * FROM SNOWFLAKE_SAMPLE_DATA.TPCH_SF1.ORDERS
Why use sources?
  • Appear in the DAG (lineage tracking)
  • Enable freshness checks
  • Single place to rename/move tables
  • Auto-documented
DAG

ref() and the DAG

ref() is how models depend on each other — dbt builds the execution graph automatically

-- models/silver/silver_orders_enriched.sql
SELECT
    o.order_id,
    o.order_date,
    o.total_price,
    c.customer_name,
    c.market_segment,
    n.nation_name
FROM {{ ref('bronze_tpch_orders') }}    o
JOIN {{ ref('bronze_tpch_customers') }} c ON o.customer_id = c.customer_id
JOIN {{ ref('bronze_tpch_nations') }}   n ON c.nation_id   = n.nation_id
-- models/gold/gold_orders.sql
SELECT
    e.order_id,
    e.order_date,
    e.customer_name,
    e.nation_name,
    e.gross_revenue,
    e.net_revenue
FROM {{ ref('silver_orders_enriched') }} e

Resulting DAG:

TPCH_SF1.ORDERS
→
bronze_tpch_orders
TPCH_SF1.CUSTOMER
→
bronze_tpch_customers
bronze_tpch_orders
→
silver_orders_enriched
→
gold_orders
🔑 ref() guarantees: correct execution order, no circular dependencies, and automatic lineage documentation.
Materializations

How Models Are Stored

Choose the right materialization for every model

view
default
Creates a SQL VIEW. Runs the query on every read. No storage cost, always fresh. Best for bronze models.
table
Creates a physical TABLE. Rebuilds from scratch every run. Best for gold that are queried heavily by BI tools.
incremental
Appends or updates only new rows. Huge performance win for large fact tables. Requires an is_incremental() filter.
ephemeral
Not materialized at all — compiled as a CTE inside the parent model. Zero warehouse objects. Use for simple intermediates.
⚙️ Set globally in dbt_project.yml or per-model with {{ config(materialized='table') }} at the top of the file.
Materializations

Incremental Models — Deep Dive

-- models/gold/gold_orders.sql
{{ config(
    materialized = 'incremental',
    unique_key   = 'order_id',
    on_schema_change = 'sync_all_columns'
) }}

SELECT
    order_id,
    customer_id,
    total_price,
    order_date
FROM {{ ref('silver_orders_enriched') }}

{% if is_incremental() %}
  -- Only process rows newer than the last run
  WHERE order_date > (
    SELECT MAX(order_date) FROM {{ this }}
  )
{% endif %}
1
First run
Creates the table with all historical data. The is_incremental() block is skipped.
2
Subsequent runs
Only processes rows after the last MAX(order_date). Merges/appends based on unique_key.
3
Full refresh
dbt run --full-refresh rebuilds from scratch when needed.
Testing

Tests — Trust Your Data

dbt has two types of tests. Both run with dbt test

Generic Tests — in schema.yml

models:
  - name: bronze_tpch_orders
    columns:
      - name: order_id
        tests:
          - unique
          - not_null
      - name: order_status
        tests:
          - accepted_values:
              values: ['O', 'F', 'P']
      - name: customer_id
        tests:
          - relationships:
              to: ref('bronze_tpch_customers')
              field: customer_id

Singular Tests — custom SQL

-- tests/assert_positive_net_revenue.sql
-- If this query returns any rows → TEST FAILS

SELECT order_id, net_price
FROM {{ ref('bronze_tpch_lineitem') }}
WHERE net_price < 0
4 built-in tests
unique, not_null, accepted_values, relationships
dbt-utils adds 20+
not_empty_string, at_least_one, expression_is_true…

Seeds

Seeds — Static Data in Git

Load CSV files directly into your warehouse as tables

seeds/country_codes.csv

country_code,country_name,region
US,United States,Americas
GB,United Kingdom,Europe
DE,Germany,Europe
MX,Mexico,Americas
JP,Japan,Asia-Pacific
# Load into warehouse
dbt seed

# Reference in a model
SELECT * FROM {{ ref('country_codes') }}
Good use cases for seeds
  • Country / currency / region lookup tables
  • Cost mappings (e.g. channel costs)
  • Marketing campaign metadata
  • Fiscal calendar definitions
⚠️ Not for large datasets. Seeds are meant for small, rarely-changing reference tables (< a few thousand rows). For large data use sources.
Snapshots

Snapshots — Track Historical Changes

Slowly Changing Dimensions (SCD Type 2) made easy

-- snapshots/customers_snapshot.sql
{% snapshot customers_snapshot %}

{{ config(
    target_database = 'ANALYTICS',
    target_schema   = 'snapshots',
    unique_key      = 'customer_id',
    strategy        = 'timestamp',
    updated_at      = 'updated_at'
) }}

SELECT * FROM {{ source('raw', 'customers_seed') }}

{% endsnapshot %}
dbt snapshot

dbt adds these columns automatically:

dbt_scd_id  — Surrogate key per record version
dbt_updated_at  — When dbt processed this row
dbt_valid_from  — When this version became active
dbt_valid_to  — When it was superseded (NULL = current)
📷 Think of snapshots as time-travel for your data. Query "what did this customer look like on March 1st?" becomes trivial.
Macros

Macros & Jinja — DRY SQL

Don't Repeat Yourself. Write logic once, reuse everywhere.

macros/cents_to_dollars.sql

{% macro cents_to_dollars(column_name) %}
    ({{ column_name }} / 100.0)::DECIMAL(10,2)
{% endmacro %}

macros/generate_surrogate_key.sql

{% macro surrogate_key(field_list) %}
    MD5(
        CONCAT_WS('|',
            {% for field in field_list %}
                COALESCE(CAST({{ field }} AS VARCHAR), '')
                {% if not loop.last %}, {% endif %}
            {% endfor %}
        )
    )
{% endmacro %}

Using macros in models:

SELECT
    order_id,
    {{ net_price('extended_price', 'discount_rate') }} AS net_price,
    {{ surrogate_key(['order_id','line_number']) }} AS sk
FROM {{ ref('bronze_tpch_lineitem') }}
Jinja essentials
{{ ... }} — expression (outputs value)
{% ... %} — statement (logic, no output)
{# ... #} — comment
{% if %} / {% for %} — control flow
{{ this }} — current model's relation
{{ env_var('VAR') }} — environment variables

  target: dev
  outputs:
    dev:
      type: snowflake
      account: xy12345.us-east-1

      account: xy12345.us-east-1
      user: DBT_SVC_ACCOUNT

❄️ Snowflake objects dbt creates
Each developer has their own schema (e.g. dbt_dev_juanpablo)
Prod writes to shared schemas (bronze, silver, gold)
dbt creates tables, views, and transient tables
Performance tips
  • Use transient_table for silver models (no fail-safe = cheaper)
  • Set query_tag to track costs per model
  • Use copy_grants=true to preserve permissions on deploys

Best Practices

Best Practices

🏗️
Layer your models
Bronze: 1-to-1 with source tables, rename/cast only.
Silver: Business logic, joins.
Gold: Final fact/dim tables for consumption.
📛
Consistent naming
bronze_, silver_, gold_ prefixes following the Medallion layers. Snake_case everywhere.
🔒
Never hardcode
Always use {{ ref() }} and {{ source() }}. Use env_var() for secrets.
🌿
One model, one concern
Keep models focused. A 500-line model with 15 joins is a future debugging nightmare.
